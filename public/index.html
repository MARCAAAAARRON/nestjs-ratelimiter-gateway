ARCHITECTURE

Four decisions that make this a gateway, not a toy.

Each one exists because the naive version breaks under real conditions — multi-instance scale, concurrent bursts, or a client that needs to know when to retry.

01 · CONCURRENCY

Atomic Lua Scripting

A separate INCR then EXPIRE leaves a window where a crash orphans a key with no TTL — rate-limited forever. incr-and-expire.lua and token-bucket.lua run as one uninterruptible EVAL, so Redis guarantees no request can interleave mid-decision.

local count = redis.call("INCR", KEYS[1]) if count == 1 then redis.call("PEXPIRE", KEYS[1], ARGV[1]) end
02 · HTTP SEMANTICS

429, Not 403

403 says "not allowed, period." 429 says "allowed, slow down" — and comes with a Retry-After header real clients use to back off automatically. A custom exception + exception filter attaches it, computed live from Redis PTTL.

HTTP/2 429 Too Many Requests retry-after: 47
03 · ALGORITHM DESIGN

Fixed Window ↔ Token Bucket

Fixed window resets hard at the boundary — a burst on either side of the reset can nearly double the effective rate. Token bucket refills continuously instead. Both are implemented, atomic, and selectable per-request via x-algorithm.

-H "x-algorithm: token-bucket"
04 · OBSERVABILITY

Prometheus + Grafana

Every allow/reject is a labeled counter (apiKey, algorithm), scraped by Prometheus and rendered on a live Grafana dashboard — the same signal a real on-call engineer would page off of.

rate_limit_rejected_total{ algorithm="token-bucket" } 1285

LOAD TEST

50 concurrent connections. 10 seconds. Zero leaks.

Benchmarked live against the Railway deployment with autocannon, one API key, RATE_LIMIT_MAX_REQUESTS=5.

Algorithm 2xx 429 Avg req/sec p50 latency p99 latency
Fixed Window 5 1,262 126.7 333 ms 2,036 ms
Token Bucket 5 1,285 129.0 332 ms 2,048 ms
▲ Exactly 5 requests admitted, every request after that correctly rejected — for both algorithms, under genuine 50-connection concurrency. This is the practical stress test for the atomic Lua fix: the naive INCR-then-EXPIRE approach has a real race window at exactly this kind of concurrent burst.
